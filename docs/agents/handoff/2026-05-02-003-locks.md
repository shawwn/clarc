# `lock.arc` — 2026-05-02

A small threading library: arc-friendly wrappers over `sb-thread` mutex
and semaphore plus a writer-fair reader-writer lock.

## Why now

Two recent changes opened the door:

- Handoff [`001`](./2026-05-02-001-setf-arc-global-ref.md) — CL place
  modifiers work on arc globals.
- Handoff [`002`](./2026-05-02-002-quasisyntax.md) — `#`...#,...` lets
  arc call CL macros that need literal-form args.

`sb-thread::with-mutex` is a CL macro that destructures `(mutex &key)`
as its first form; before quasisyntax there was no clean way to call
it from arc (ac would compile the binding spec as a function call).

## API

```arc
;; Mutex
(make-mutex)             ; or (make-mutex 'some-name)
(w/mutex m body...)      ; macro: hold m for body, release on any exit

;; Semaphore (counting)
(make-semaphore)         ; default count 0
(make-semaphore 1 'name) ; initial count, optional name
(sem-wait s)             ; decrement (blocks at 0)
(sem-post s)             ; increment (wakes one waiter)

;; Reader-writer lock
(make-rwlock 'name)
(w/read-lock rw body...)   ; concurrent with other readers
(w/write-lock rw body...)  ; exclusive
```

`make-mutex` / `make-semaphore` / `make-rwlock` accept arc symbols for
`name` and coerce to a string for sb-thread's `:name` slot, which
requires `(or simple-string null)`.

## `w/mutex` via quasisyntax

```arc
(mac w/mutex (m . body)
  `(#'sb-thread::with-mutex #`(#,,m) ,@body))
```

Two layers of quoting:

- Outer arc backquote — substitutes the macro's `m` and `body` args at
  expansion time.
- Inner `#`(#,,m)` — produces a literal CL form `(<m-value>)` for the
  `(mutex &key)` slot that `sb-thread::with-mutex` destructures.

The body forms splice via plain `,@body` and ride out as ordinary arc
forms; ac compiles them when it processes the resulting macro-call form
(`((function ...) args)` triggers ac line 453's mapcar). No
`#,(do ,@body)` wrapping needed because the body isn't *inside* the
quasisyntax — only the binding slot is.

## RW lock — turnstile pattern

The lock is a 4-field table:

| Field | Type | Role |
|---|---|---|
| `m` | mutex | protects `readers` |
| `readers` | int | in-flight reader count |
| `ws` | semaphore (1) | "writer slot" / "no readers active" permit |
| `turnstile` | semaphore (1) | gate on *arrival*; readers grab-and-release, writers hold |

### Reader path

```arc
(sem-wait turnstile) (sem-post turnstile)   ; pass through
(w/mutex m
  (when (is readers 0) (sem-wait ws))       ; first reader claims slot
  (++ readers))
;; ... body ...
(w/mutex m
  (-- readers)
  (when (is readers 0) (sem-post ws)))      ; last reader releases
```

### Writer path

```arc
(sem-wait turnstile)                         ; block all new arrivals
(sem-wait ws)                                ; wait for in-flight readers
;; ... body ...
(sem-post ws) (sem-post turnstile)
```

### Why writers don't starve

The turnstile is the key invariant. A reader holds it for ~zero time
(grab, immediately release on entry). A writer holds it for the *entire
wait + critical section*. From the moment a writer holds the turnstile:

- No new reader can pass through (they block at `sem-wait turnstile`).
- The set of in-flight readers is fixed and can only shrink.
- Each completing reader decrements the count; when it hits zero the
  last reader posts `ws`.
- The writer's `sem-wait ws` returns; the writer enters its critical
  section.

Worst-case writer wait time = the longest currently-in-flight reader's
critical-section duration. **Bounded by reader work, not reader arrival
rate.** A flood of new readers can't keep `readers` above zero once a
writer holds the turnstile.

### Mild reader-starvability

A stream of writers can hold the turnstile in sequence, queueing
readers indefinitely. This is the standard writer-preference tradeoff;
strict fairness would require an explicit FIFO (mutex + condvar)
instead of two semaphores. Documented in `lock.arc` so a future tuner
knows where the policy lives.

## Layout note

Not auto-loaded from [`libs.arc`](../../../libs.arc). Anything that
wants the lock primitives loads `lock.arc` explicitly. Reason for the
opt-in: there's no consumer in the standard load path yet, and adding
a forced load before any callers exist just bakes a dependency for no
gain. [`test.arc`](../../../test.arc) does its own `(load "lock.arc")`
near the top of the file so the lock test set runs.

## What's not here

- **Condition variables** (`sb-thread:make-waitqueue` /
  `condition-wait`). Useful for predicate-waits ("wait until queue
  non-empty") and for fair RW locks. Skipped because the current
  use cases are RW-shaped.
- **Reentrant mutex**. `sb-thread::make-mutex` produces a
  non-recursive lock; double-acquire from the same thread deadlocks.
  If reentrancy is ever wanted, `sb-thread::make-mutex` accepts no
  recursive option but you can build reentrancy on top with a
  per-thread owner check.
- **Lock-free counters**. See handoff `001`'s tail discussion of
  `atomic-incf` — needs a different storage strategy.
