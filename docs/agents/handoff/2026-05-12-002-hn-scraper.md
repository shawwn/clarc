# Handoff: HN scraper (2026-05-12)

## What landed

Two new files + small extensions:

- `scrape.arc` (525 lines) -- HTTP via curl, HTML parser for HN's
  `<tr class="athing comtr">` rows, top-level `scrape!` and
  `import-scrape!` entry points.
- `json.arc` (212 lines) -- standalone JSON encoder + decoder used by
  the scraper.  No external deps.
- `test.arc` -- 9 new `define-test json-*` blocks (encode/decode
  primitives, escapes, arrays, objects, roundtrip).  Suite went from
  241 to 291 passing, 0 failing.
- `arc0.lisp:659-666` -- one-line change: `pipe-from` now uses
  `:external-format :latin-1`.  Without this, subprocess stdout was
  decoded as UTF-8 (SBCL default), then writing those chars to
  Latin-1 output files (the `outfile` default) failed on any
  non-Latin-1 codepoint like U+201C left-double-quote.  Latin-1
  byte-pass-through everywhere keeps `curl | parse | save` clean.
- `arc/scrape/verify-flags.arc` -- assertion script that re-fetches
  the 4 context items from `docs/agents/prompt.txt` and asserts each
  named comment has the expected `dead`/`flagged`/`collapsed`
  combination.  Run with `./sharc arc/scrape/verify-flags.arc`.

## Usage

```sh
./sharc                          # then at the REPL:
arc> (load "scrape.arc")
arc> (scrape!)                   ; full run, ~60 items (default limit)
arc> (scrape! nil 5)             ; dev: just the top 5
arc> (scrape! t)                 ; force re-scrape even if cached
```

Then to mirror HN on the local News server:

```arc
arc> (load "news.arc")
arc> (load "scrape.arc")
arc> (import-scrape!)            ; populates stories*/items*/profs*
arc> (asv 8080)                  ; server now serves HN's front page
```

## Data layout

Everything writes under `arc/scrape/`:

- `cookies.txt` -- curl cookie jar (HN session for `hnscraper`).
  Login is automatic via `ensure-login` whenever the session is
  invalid; credentials are hardcoded as defaults to `hn-login`.
- `last-fetch.lisp` -- table of `id -> unix-seconds`.  An item id
  recently fetched (< `scrape-refetch-secs*`, default 3600) is
  skipped on the next pass.  User entries use the key `(sym "u/" id)`.
- `front.json` -- last scrape's ordered story list: `[{id, page, rank}, ...]`.
  Read by `import-scrape!` to reconstruct HN's order.
- `item/{id}.json` -- merged story + comment tree for one item.
- `user/{id}.json` -- one user profile (from Firebase API).

The per-item JSON shape:

```json
{ "story": {"id":..., "type":"story", "by":..., "title":..., "score":...,
            "url":..., "text":..., "time":..., "dead":..., "fetched_at":...},
  "comments": [ {"id":..., "by":..., "time":..., "parent":..., "indent":...,
                 "text":..., "dead":..., "flagged":..., "collapsed":...,
                 "deleted":..., "descendants":...},
                ... ] }
```

## Key decisions

### 1. Front-page ordering from the API, not HTML

The HN Firebase API's `/v0/topstories.json` returns up to 500 ranked
story ids in one call.  That's authoritative and replaces parsing
`/news` + `/news?p=2`.

The original spec said "scrape the front page ourselves", but the
user later clarified that the API is more faithful at any given
moment, and per-item HTML scraping (which is still needed for
dead/flagged/collapsed comment info) covers the use-case.  The old
HTML-front-page parser code was deleted.

### 2. Per-item HTML is the only source for comment status

Firebase's `/v0/item/{id}.json` exposes `dead: true` for dead
comments but **not** `[flagged]`-only state or `[collapsed]` state.
So per-item HTML is mandatory for the scraper's stated purpose.

The user was explicit: **do not use the API for items/comments**.
Only for users.  `fetch-user` is the only API call besides
`topstories.json`.

### 3. Crawl-delay = 3s, overriding robots.txt's 30s

`hnscraper`'s About page invites contact rather than ban, and the
user explicitly authorized the faster rate ("you can also scrape
every few seconds instead of every 30s.  Maybe 3s").  Hard-coded as
`scrape-crawl-delay*` at the top of scrape.arc with a comment
explaining the deviation.  Revert to 30 if HN ops complains.

There's no `(fetch-robots-delay)` anymore -- it was removed in the
cleanup pass once the value was hardcoded.

### 4. Per-comment HTML scoping (perf)

The first version of `parse-comments` called `posmatch` on the full
2MB html for each of ~1000 comment-row fields.  That's O(C*N) and
hung for minutes on a real item page.

The fix is in `parse-comments`: walk the full html once to collect
the byte offsets of every `<tr class="athing comtr` anchor, then for
each comment cut the substring between consecutive anchors and pass
**that** (~1KB per row) to `parse-comment-row`.  All subsequent
posmatches scan within the row's substring.  Drops parse time from
"hangs >5min" to <1s for a 2MB item page.

### 5. Parent-id derivation from indent stack

HN's HTML omits the `parent` id on comment rows (it's only on the
permalink "context" view).  But comments are rendered in DFS order
with `<td class="ind" indent="N">` carrying the depth.  So we keep
an `indent-stack` table mapping `level -> id-of-most-recent-row`,
and a comment at indent N has parent = `indent-stack[N-1]` (or the
story id if N=0).  Wipe deeper-level entries each time we descend.

### 6. Deletion-aware comment merge

When refetching an item, `build-item-json` merges the new comments
with the previously-saved set: if an id existed before but is
missing from the new fetch, it's kept with `deleted: true`.  Order
of merged output is new-fetch-first, then the now-deleted entries
appended at the end.  The user asked for this explicitly.

### 7. JSON-decoded `type` is a *string*, not a symbol

The encoder turns Arc symbols into JSON strings; on decode they come
back as strings.  News's `displayfn*` table is keyed by symbols, so
`(displayfn* "story")` returns nil and rendering crashes with the
unhelpful "Function call on non-function: NIL".  Fix is in
`import-scraped-story`: `'type (sym (or s!type "story"))`.

`import-scraped-comment` hardcodes `'type 'comment`, so it doesn't
have the same problem.

## Gotchas discovered along the way

### Arc macro-vs-local-var shadowing

A local variable named `out` or `row` silently breaks anywhere it's
used in head-of-form position, because Arc's compile-time
macroexpansion runs on `(var ...)` calls regardless of local
bindings.  `out` is `(mac out (expr) ... (eval expr) ...)` in
arc.arc; `row` is `(mac row args ... `(tr ...))` in html.arc.
Symptoms:

- `out` shadowing: compile-time `(eval some-local)` -> "Unbound
  variable" error during loading.
- `row` shadowing: runtime emits stray `<tr><td>fieldname</td></tr>`
  per access, because `(row 'indent)` expands to `(tr (td 'indent))`.

Variables in scrape.arc were renamed `out -> rec` and `row -> c` /
`entry`.  Other shadowing-prone names to avoid: `tr`, `td`, `tdr`,
`trtd`, `tab`, `form`, `center`, `link`, `but`, `input`, `br`,
`tag`.

### Bracket-lambda + `!` ssyntax footgun

`[_!dead]` does **not** mean `(fn (_) (_ 'dead))`.  Bracket lambdas
always wrap the body in a call form, so it expands to
`(fn (_) (_!dead))` = `(fn (_) ((_ 'dead)))` -- it calls the
*result* as a function, which blows up if it's nil/t/anything
non-callable.

Workaround: use `(fn (x) x!dead)` or `[_ 'dead]`.  Multi-token
bracket bodies are fine: `[item _!id]` -> `(fn (_) (item _!id))` ->
`(fn (_) (item (_ 'id)))` works.

### `(/ a b)` returns a rational, not an int

`(mod (/ 31 16) 16)` -> `(mod 31/16 16)` -> still a rational, not
indexable into a string.  Use `(trunc (/ n 16))` for integer
division.  Bit me in `json-write-string`'s \uXXXX path.

### Arc's `loop` and `cond`

- `cond` does **not** exist in Arc.  Use `(if t1 r1 t2 r2 ... default)`.
- `loop` is `(loop init test update body)`, not `(loop ((forever))
  ...)`.  Use `(while t ...)` or `(catch (while t ... (throw)))` for
  early exit instead.
- `(return X)` doesn't exist.  Use `(catch ... (throw X))`.

### Arc's `let` is single-binding

`(let a 1 b 2 ...)` is `(let a 1 b 2 ...)` parsed as binding `a` to
`1` and `b` to `2`... wait no, `let` only takes one binding.  For
multiple, use `with` (parallel) or `withs` (sequential).  Bit me in
`parse-item-page` where I wrote `(let story (...) comments (...) ...)`.

### `writefile` Lisp-escapes string values

`(writefile some-html-string path)` writes `"escaped contents"`, not
raw bytes.  For raw output use `w/outfile o path (disp val o)`.  My
JSON path uses `w/outfile` directly and is fine; HTML caching was
never necessary so I removed it.

## Notable concerns / future work

- **User-fetch throughput.**  Each user is a `curl` subprocess
  (~250ms-1s including SSL handshake).  A full 60-item scrape can
  produce 1000+ unique users -> 5-20 min.  Worth parallelizing via
  `xargs -P` or a coroutine pool.
- **News id collisions.**  `import-scrape!` writes HN ids directly
  into `arc/news/story/`.  If you have local items at the same ids,
  they'll be overwritten.  Fine on a fresh install, dangerous
  otherwise.  Could namespace under a separate `arc/news/hn/`
  hierarchy if needed.
- **`maxid*` gap.**  After import, `maxid*` jumps to ~48M.  The next
  locally-created item starts from there, which makes id sequencing
  visually weird but functionally fine.
- **No item-level dead flag.**  We rely on `topstories.json`, which
  already filters out dead stories.  Front-page-level `[dead]` /
  `[flagged]` story status (rare; mostly affects newest/show pages)
  is not captured.  If you need that, you'd have to re-add an HTML
  front-page parser for `/news` (deleted in this pass), or shift
  source to `/newest`.
